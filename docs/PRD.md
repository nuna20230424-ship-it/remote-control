# PRD — STB 리모컨 학습 에이전트 (remotectl)

- 문서 버전: v1.0
- 작성일: 2026-07-07
- 대상 릴리스: 1차 릴리스 (Mock 드라이버로 완결)
- 패키지명: `remotectl`

---

## 1. 개요 (Summary)

STB 리모컨 학습 에이전트는 셋탑박스(STB)를 리모컨으로 조작하면서 화면 전이를 관찰·학습하고,
그 관찰을 **네비게이션 맵(상태 그래프)** 으로 구축한 뒤, 자연어 목표(예: "넷플릭스 켜줘")를 받아
**눌러야 할 버튼 시퀀스를 스스로 계획·실행**하는 목표기반 에이전트다.

핵심 루프는 세 단계로 나뉜다.

1. **학습(Explore)**: 버튼 입력 → 화면 전이를 관찰하여 상태·간선(edge)을 수집한다.
2. **맵 구축(Build)**: 관찰을 상태 그래프로 누적하고, 상태를 안정적으로 식별(state id)한다.
3. **목표기반 실행(Plan & Act)**: 목표를 목표 상태로 해석하고, 맵 위에서 최단 버튼 시퀀스를
   계획한 뒤 실행하며, 실제 전이가 예측과 다르면 재계획한다.

실제 STB 제어는 사내망의 기존 **remote-MCP**(172.16.x HTTP)를 재사용하되, 엔드포인트가 미확정이므로
`RemoteDriver` 추상 인터페이스로 감싼다. 화면 상태 판정은 detection-mcp/VLM을 `ScreenSense`
인터페이스로 추상화한다. 1차 릴리스는 **Mock 드라이버 + Mock 센스로 엔드투엔드 완결**되며,
실제 HTTP 클라이언트는 동일 인터페이스 구현체로 무중단 교체 가능하도록 설계한다.

---

## 2. 목표 (Goals)

- G1. 리모컨 버튼 입력과 그에 따른 화면 전이를 관찰하여 재사용 가능한 네비게이션 맵을 자동 구축한다.
- G2. 구축된 맵을 조회·시각화하여 상태·전이·커버리지를 사람이 이해할 수 있게 한다.
- G3. 자연어 목표를 목표 상태로 해석하고, 맵 위에서 버튼 시퀀스를 계획·실행하여 목표에 도달한다.
- G4. 실제 STB(remote-MCP) 없이도 Mock 드라이버/센스만으로 전체 파이프라인이 동작·테스트되게 한다.
- G5. 드라이버(RemoteDriver)와 화면 판정(ScreenSense)을 인터페이스로 분리하여, 사업자·단말·백엔드
  교체 시 코어 로직 변경 없이 어댑터만 갈아끼우게 한다.

### 비목표 (Non-Goals, 명시)

- 특정 STB 펌웨어/UI에 하드코딩된 스크립트 기반 자동화 (범용 맵 학습으로 대체).
- 픽셀 단위 UI 검증/회귀 테스트 (본 제품은 네비게이션·도달성에 집중).
- 실시간 영상 스트리밍 품질 측정.

## 3. 페르소나 & 니즈 (Personas & Needs)

### P1. QA 자동화 엔지니어 (Primary)
- 니즈: 새 STB 모델이 들어올 때마다 리모컨 조작 시나리오를 수작업 스크립팅하지 않고, 에이전트가
  UI를 탐색·학습해 맵을 만들어주길 원한다. "넷플릭스 켜줘" 같은 목표를 던지면 알아서 도달해야 한다.
- 성공의 모습: 단말별 스크립트 유지보수 비용 제거, 목표기반 실행으로 테스트 케이스 작성 시간 단축.

### P2. 테스트 리드 / QA 매니저
- 니즈: 어떤 화면·기능이 학습되었는지(커버리지), 어디로 도달 가능한지 한눈에 보고 싶다. 실패한
  목표 실행의 원인(어느 상태에서 전이가 예측과 달랐는지)을 추적하고 싶다.
- 성공의 모습: 맵 조회 대시보드에서 커버리지·미탐색 영역·실행 성공률을 확인.

### P3. 플랫폼/도구 개발자 (본 제품 개발자)
- 니즈: remote-MCP 엔드포인트가 확정되기 전에도 개발·테스트를 진행하고, 확정 후 어댑터만 교체하고
  싶다. detection-mcp/VLM 백엔드도 동일하게 갈아끼우고 싶다.
- 성공의 모습: 인터페이스 계약(RemoteDriver/ScreenSense)이 안정적이고, Mock↔실물 교체가 무중단.

## 4. 유스케이스 (Use Cases)

### UC-1. 학습 세션 (Explore & Build)
- 흐름: 사용자가 시작 상태(또는 홈)에서 학습 세션을 시작한다 → 에이전트가 후보 버튼 집합
  (예: HOME, UP/DOWN/LEFT/RIGHT, OK, BACK, 숫자, 앱 단축키)을 정책(예: 미탐색 간선 우선 BFS)에
  따라 선택한다 → `RemoteDriver.press(button)`로 입력하고, 안정화 대기 후 `ScreenSense.observe()`로
  현재 화면을 관찰한다 → 관찰을 상태로 식별(기존 상태 매칭 또는 신규 생성)하고 `(from, button, to)`
  간선을 맵에 기록한다 → 예산(스텝 수/시간) 또는 커버리지 목표 도달 시 종료한다.
- 성공 기준: 세션 종료 시 방문 상태 수·간선 수·미탐색 간선 수가 집계되고, 맵이 영속화(저장)된다.
- 실패/엣지: BACK 루프·순환, 판정 불확실(저신뢰) 상태는 재관찰 또는 별도 표시.

### UC-2. 맵 조회 (Inspect Map)
- 흐름: 사용자가 REST API 또는 대시보드로 현재 맵을 조회한다 → 상태 목록(id, 라벨, 대표 관찰),
  전이 목록(from→to via button), 커버리지 지표를 확인한다 → 특정 상태에서 도달 가능한 상태/경로를
  조회한다.
- 성공 기준: 상태·전이가 정확히 반영되고, 대시보드에서 그래프 형태로 확인 가능.
- 엣지: 빈 맵(학습 전) 시 명확한 안내, 대규모 맵의 페이지네이션/필터.

### UC-3. 목표기반 실행 (Plan & Act toward Goal)
- 흐름: 사용자가 자연어 목표("넷플릭스 켜줘")를 제출한다 → 에이전트가 목표를 목표 상태
  (예: `app:netflix`)로 해석한다 → 현재 상태를 관찰로 확정한다 → 맵 위에서 현재 상태→목표 상태
  최단 버튼 시퀀스를 계획(networkx 최단경로)한다 → 버튼을 하나씩 실행하고 매 스텝 관찰로 실제 전이가
  계획과 일치하는지 검증한다 → 불일치 시 현재 상태 기준으로 재계획한다 → 목표 상태 도달 또는 재시도
  예산 소진 시 종료한다.
- 성공 기준: 목표 상태에 도달하면 성공, 실행 경로(버튼 시퀀스)와 스텝별 관찰이 결과로 반환된다.
- 실패/엣지: 목표 상태가 맵에 없음(미학습) → 안내/부분 탐색 제안. 계획 불가(도달 불가) → 사유 반환.
  실행 중 예측 이탈 반복 → 재계획 한도 초과로 실패 처리.

## 5. 스코프 (Scope)

### 5.1 In Scope (1차 릴리스 — Mock으로 완결)
- `RemoteDriver` 추상 인터페이스 + `MockRemoteDriver`(결정적 상태전이 시뮬) + `RemoteMcpClient`(httpx HTTP) 스텁.
- `ScreenSense` 추상 인터페이스 + `MockScreenSense`(결정적 화면 라벨 반환). 실 detection-mcp/VLM 어댑터는 인터페이스 자리만 확보.
- 네비게이션 맵: Pydantic v2 데이터모델(State/Edge/Observation) + networkx 그래프 저장/로드/영속화(JSON).
- 학습 세션 엔진: 탐색 정책(미탐색 간선 우선 BFS), 상태 식별(관찰→state id), 예산/커버리지 종료 조건.
- 계획·실행 엔진: 자연어 목표→목표 상태 매핑(1차: 규칙/키워드 매핑), 최단경로 계획, 스텝별 검증·재계획.
- FastAPI REST: 학습 시작/상태, 맵 조회, 목표 실행 엔드포인트 + 정적 대시보드 서빙.
- 대시보드: 맵 그래프 시각화, 커버리지·실행 결과 표시(정적 HTML/JS로 자족).
- pytest: Mock 기반 엔드투엔드 테스트(학습→맵→목표실행 전 경로), 인터페이스 계약 테스트.

### 5.2 Out of Scope (1차 제외)
- 실 remote-MCP HTTP 엔드포인트 실연동(엔드포인트 미확정 — 클라이언트는 스텁/인터페이스만).
- 실 detection-mcp/VLM 판정 실연동(어댑터 골격만, 실제 모델 호출은 후속).
- 자연어 목표의 LLM 기반 해석(1차는 규칙 기반; LLM 해석은 후속 확장).
- 멀티 STB 동시 세션 오케스트레이션, 인증/권한(RBAC), 대규모 맵 분산 저장.
- 사업자별 UI 라벨 온톨로지 표준화(후속).

## 6. 성공지표 (Success Metrics)

- M1. Mock 엔드투엔드: 학습→맵구축→목표실행 파이프라인이 실물 없이 100% 통과(pytest 그린).
- M2. 목표 도달률: Mock 시뮬 환경에서 맵에 학습된 목표에 대해 실행 성공률 >= 95%.
- M3. 재계획 견고성: 실행 중 예측 이탈(전이 불일치) 주입 시에도 재계획으로 목표 도달률 >= 90%.
- M4. 학습 커버리지: 정해진 스텝 예산 내 도달가능 상태의 >= 90%를 방문(Mock 맵 기준).
- M5. 교체성: RemoteDriver/ScreenSense 구현체 교체 시 코어 엔진 코드 변경 0줄(인터페이스만으로 성립).
- M6. 계획 정확성: 맵상 최단경로 대비 실행 버튼 수 오버헤드 중앙값 <= 1스텝.

## 7. 리스크 (Risks)

- R1. remote-MCP 엔드포인트 미확정 → `RemoteDriver` 인터페이스가 실제 계약과 어긋날 수 있음.
  완화: 최소 공통 동작(press/observe/reset)만 계약화, 실물 확정 시 어댑터 내부에서 흡수.
- R2. 상태 식별 불안정(같은 화면이 다른 관찰로, 다른 화면이 같은 관찰로) → 맵 오염.
  완화: 관찰 정규화·신뢰도 임계값·상태 병합/분리 규칙, 저신뢰 관찰 재관찰.
- R3. 화면 전이 비결정성(로딩·애니메이션·네트워크) → 관찰 타이밍 이슈.
  완화: 안정화 대기(settle) + 재관찰, 전이 확률/재시도 기록.
- R4. 자연어 목표 해석 한계(규칙 기반) → 목표 상태 매핑 실패.
  완화: 키워드/별칭 사전 + 미매핑 시 명확한 실패 사유, 후속 LLM 해석 확장 여지.
- R5. 맵 미학습 영역으로의 목표 → 계획 불가.
  완화: 미학습 감지 시 부분 탐색 제안/온디맨드 탐색 훅.
- R6. VLM/detection 판정 비용·지연(실연동 시) → 학습 세션 속도 저하.
  완화: 관찰 캐싱, 저비용 mock/휴리스틱 선판정 후 필요 시 VLM 승격(후속).
- R7. 대시보드/REST가 자족(오프라인) 요구를 위반(외부 CDN 의존) → 사내망 미동작.
  완화: 정적 자산 인라인/자족 번들.
